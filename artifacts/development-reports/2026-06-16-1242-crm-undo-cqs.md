# Relatório — Desfazer o CQS (ReadService) e organizar os data records em `domain.crm.service.data`

- **Data:** 2026-06-16
- **Branch:** `refactor/crm-undo-cqs` (a partir de `develop`)
- **Versão:** **0.14.2** (refactor interno → PATCH; bump por mudança, sem release note)
- **Tipo:** refatoração interna, **sem mudança de comportamento nem de contrato** (JSON idêntico; os
  testes de integração de leitura passam inalterados, como rede de segurança).

## 1. O que foi feito (Regra Zero — decisão do dono)

A "opção B" anterior (montagem de leitura em `application.read`, um `*ReadService` por área, e os
records de leitura apagados do domínio) foi considerada **overengineering / CQS desnecessário**. Foi
**revertida**:

- **Sem CQS.** As leituras voltaram para os **Application Services do domínio** — `LeadService` e
  `OpportunityService` fazem **comando E consulta**. As transições (`qualify/markLost/reassign/
  recordInteraction`) voltam a **retornar `LeadDetail`** (montado dentro da própria transação de
  escrita; sem re-leitura). **`application.read` foi apagado** (2 services + o teste).
- **`domain.crm.service.data`** passa a concentrar **todos os data records do serviço**:
  - inputs: `RegisterLeadCommand`, `RecordInteractionCommand`, `CreateOpportunityCommand`,
    `ReferenceCommand`, `LeadSearchCriteria`, `OpportunitySearchCriteria` (movidos de `domain.crm.dto`,
    que foi **apagado**);
  - read models: `LeadListItem`, `LeadDetail` (+ aninhados), `PendingLead`, `LeadIndicators`
    (+ aninhados), `OpportunityListItem`, `Responsible` — trazidos dos antigos
    `application.api.dto.*Response`, **renomeados sem o sufixo `Response`** (termo de entrega não fica
    no domínio).
- **Controllers** voltam a chamar o domain service e **retornam o read model diretamente** (paginado
  via `PageResponse`); escritas retornam `LeadDetail`. `application.api.dto` mantém só os **requests** e
  as respostas triviais de create (`LeadResponse`, `OpportunityResponse`). `controllersMustNotAccess
  Repositories` segue válido (controller → service, nunca repo).
- **`LeadIndicatorQueries`** continua devolvendo `Map`s ordenados (sem records de view no domínio).

## 2. Regras de design fixadas no CLAUDE.md

- **§5.1/§5.4/§6:** um **Application Service por área faz comando + leitura** (sem CQS, sem
  `application.read`); os **data records do serviço (inputs + read models) vivem em
  `<area>.service.data`**; o controller retorna o read model do domínio (entidade nunca vaza); **não**
  duplicar com `*Response` no `application.api.dto`; **sem pacote `dto` no domínio**.
- **§14** (mantido): release note só no fim de uma entrega completa; bump de versão por mudança.
