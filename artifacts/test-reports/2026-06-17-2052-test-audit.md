# FKERP — Test Audit & Coverage Report

- **Date:** 2026-06-17
- **Scope:** full test suite (backend + frontend + E2E), realness assessment, coverage, gaps fixed, TDD adoption
- **Prepared by:** engineering (agent-assisted)
- **Verdict:** the suite is **real and contract-asserting**, not coverage-theater. Backend coverage is **96.8%
  instructions / 96.9% lines / 84.2% branches**. Frontend **application logic (`.ts`) is 85.6%**; the headline
  blended percentage (38%) is **template-skewed** — proven below by splitting `.ts` vs `.html`. Genuine
  under-tested files were found and closed via TDD this cycle (frontend specs 167 → **199 tests**).

---

## 1. Methodology

Each test was reviewed against two questions (per CLAUDE.md §13):

1. **Is it real?** Does it assert the **contract** — the full error body `{code, message, fields}`, the i18n
   `code`, the offending field, and the success **shape** — rather than only an HTTP status or a truthy value?
2. **Does it cover the sad paths?** Not just the happy path, but the plausible failures: `401` (unauthenticated),
   `403` (missing scope / not visible), `400` (bean validation), `404` (not found), `409` (conflict/duplicate),
   `422` (business rule), boundary values, and idempotency/visibility edges.

Coverage was then measured objectively with **JaCoCo** (backend) and **vitest v8** (frontend), both newly
wired into the build. Coverage is used as a **signal to find gaps**, never as the goal.

### 3.2 Frontend — vitest v8 (`frontend/coverage/`)

The blended headline (**statements 38.3% · lines 47.5%**) is **template-skewed**. Splitting the
`coverage-final.json` by file type proves it (statement coverage):

| File type | Coverage | Counted statements |
|---|---|---|
| **`.ts` (application logic)** | **85.6%** | 1138 / 1329 |
| **`.html` (component templates)** | **16.0%** | 405 / 2539 |
| Blended (what the headline shows) | 38.3% | 1543 / 3868 |

Reading: the **templates dominate the denominator** (2539 of 3868 statements) and are barely executed, because
component specs assert **class behavior and key DOM states** rather than rendering every template branch. The
**`.ts` logic — where the behavior lives — is 85.6%**: core services/guards/interceptor at or near 100%, the
list/indicator/detail components 80–96%. The **security infrastructure is tested** (CLAUDE.md §13): the auth
interceptor's **401 → refresh → replay**, the route guards, and the unsaved-changes `CanDeactivate` guard.

Spec/setup/bootstrap files (`*.spec.ts`, `main.ts`, `*.config.ts`) are **correctly excluded** from the
denominator (verified). Chasing the blended number by rendering every template state would be coverage-theater;
the honest position is that **logic is well-covered and template-state DOM coverage is the next investment**,
adopted as a TDD habit for new components (§5).

## 4. Realness assessment (are these real tests?)

**Yes.** Representative evidence by layer:

- **Backend API integration** asserts the **full contract**, not just status. Examples:
  `ProposalLifecycleApiIntegrationTest` asserts `422` **with** `$.code = proposal.discount-invalid` /
  `proposal.no-items` / `proposal.total-required`; visibility tests assert `403 proposal.access-denied`,
  finance/`401`/diretor-no-scope; `ProposalCreationApiIntegrationTest.exposesOnlyCommercialOfferFields` asserts
  the **exact JSON key set** (no Sale/Booking/Financial leakage). Every validation layer (§5.5) has a rejecting
  test with the right status **and** body.
- **Domain unit** asserts behavior and invariants: `ProposalTotalsTest` proves subtotal/discount math, the
  **total-never-negative** cap, the Draft-only guard, and the submit-for-review preconditions — each sad path is
  a distinct test.
- **Architecture** tests enforce the real boundaries (`domain ↛ application/infra`, no `@Data`/`@Setter` on
  entities, no field injection, exceptions live with their domain) and **mapping completeness** (every
  `DomainException` has an HTTP status) — these fail the build if violated.
- **Frontend** specs assert visible state and the security flows (interceptor refresh/replay, guards redirect),
  service request contracts (method/URL/body/params), and component error/empty handling — not just "it renders".
- **E2E** drives real journeys in a browser against an isolated, ephemeral Postgres stack (CORS, same-origin
  proxy, port wiring), including the unsaved-changes-via-shortcut warning and the proposal discount→total→submit
  flow.

**Anti-patterns searched for and not found:** assertions on only `status()` with no body; `expect(true)`-style
filler; tests with no assertions; disabled/`skip`ped tests hiding failures; gates weakened to pass.

## 6. TDD adopted going forward (standing decision)

From here on (owner mandate, recorded in project memory): **write the failing test first, then the
implementation**, for every behavior, covering the happy path **and all plausible sad paths**, in **all
categories**. Formalized in CLAUDE.md §13. Concretely for new work:

- New endpoint → integration test asserting `{code,message,fields}` + the matrix (401/403/400/404/409/422) before
  the controller exists.
- New component → spec rendering the real **loading / empty / error / permission-denied / success** states
  (this is the agreed lever to raise frontend template coverage honestly).
- New guard/interceptor/service → spec before wiring.

## 7. How to run coverage

```bash
# Backend — JaCoCo HTML + CSV at backend/target/site/jacoco/
cd backend && ./mvnw verify
#   open target/site/jacoco/index.html

# Frontend — vitest v8 text + lcov at frontend/coverage/
cd frontend && npx ng test --watch=false --coverage

# E2E (isolated stack, ports 4201)
cd frontend && npm run e2e:up && npm run e2e && npm run e2e:down
```
